## 1. Product overview

### 1.1 What is Winkly?

Winkly is a **multi-mode social and planning app** that combines:

- **Romance** — Dating: discover, match (mutual like), chat 1:1, plan dates.
- **Friends** — Platonic connections: match (mutual follow), 1:1 and **group chats**, invite contacts, plan meetups and events.
- **Business** — Professional discovery (LinkedIn + Instagram style): search/filter by role, company, industry; 1:1 and group chats; plan business meetings and events.
- **Events** — Create and discover events; participate; event chats; integrate with Planner.

All modes share a **unified Planner** (dates, meetups, business meetings, events) and an **AI concierge** that prioritizes Winkly events and business offers first, falling back to external suggestions only when needed.

### 1.2 Account types

| Type         | Description                                                   | Modes available                                    |
| ------------ | ------------------------------------------------------------- | -------------------------------------------------- |
| **Personal** | One user, multiple sub-profiles (Romance, Friends, Business). | Romance, Friends, Business, Events (Events always) |
| **Business** | Company/brand account.                                        | Business, Events                                   |

- **Personal**: User creates **sub-profiles** per mode (Romance, Friends, Business). Each has its own bio, photos, interests, mode-specific fields.
- **Business**: Single **profiles_business** (business name, location, bio, logo, social links). No Romance/Friends sub-profiles.
- **Events** mode is available to all users regardless of account type.
- **Dual profiles, one email:** A user may hold both a Personal core profile and a Business profile under the same auth account. **Active account type** (`users.account_type` + auth metadata) controls which modes and onboarding path apply; switching or creating the other type is available in **Account & Identity** / **Profile Settings** (`lib/account/accountTypeSwitch.ts`). Pre-signup, Get Started → Sign up pre-selects the chosen type with an inline switch link.

### 1.3 Core product principles

1. **Identity Firewall** — Data and UI are strictly separated by mode. No cross-mode leakage (e.g. Romance data never shown in Business).
2. **AI prioritizes Winkly** — AI suggests Winkly events and business offers first; external sources only as fallback.
3. **Single app, multiple personas** — One account can switch between modes; each mode has its own discover, chats, and planner context.

### 3.5 Security posture

- **Disclosure and practices:** **SECURITY.md** (repository root) — how to report issues, `auth-redirect` and JWT, RLS, rate limits, dependencies, and client bundle hygiene.
- **Row Level Security:** The mobile app uses the **anon** key; all data access must be enforced with **RLS** and tested per table / policy.

- **Repository visibility:** A public repo increases schema and flow visibility; use a **private** repository if the product is not intended to be open source.

| `AI_MAX_CONTEXT_CHARS`                           | Optional max serialized request-context size (default **24000**); larger requests return HTTP **413**.                   |

If Redis secrets are unset, rate limiting is skipped (dev-friendly); caching is inactive.

**Server-side AI access enforcement (source of truth — client gate is UX only)**

The gateway re-checks AI access for every **billable task** (`chat_topics`, `planner_theme_plans`, `event_suggest`, `plan`, `concierge`, `winkly_plan`, `match_agent`, `match_bridge`) regardless of client behavior, because client checks (`lib/ai/aiFeatureGate.ts`) are trivially bypassable with a valid session token. Order of checks: global kill switch → per-tier disable → **tier access matrix** → cost guards → rate limit.

- **Tier access matrix** (`TASK_MIN_TIER` in `ai-gateway`): mirrors the client gate. **Free = no AI** (all billable tasks denied). **Super+** = `chat_topics`, `planner_theme_plans`, `event_suggest`, `plan`, `winkly_plan`, `match_agent`. **Premium+** = `concierge`, `match_bridge` (full concierge). Denied requests return HTTP **403** with `code: "ai_tier_required"`, `required_tier`, and `upgrade_to` (`super`/`premium`) for client upsell.

**Cost guards (max tokens / input size per request)**

- **Output-token ceiling**: every LLM call (Anthropic/Gemini/OpenAI) clamps its output budget via `capOutputTokens()` to `AI_MAX_OUTPUT_TOKENS` (default 2048).
- **Input guard**: requests whose serialized allowlisted context exceeds `AI_MAX_CONTEXT_CHARS` are rejected with HTTP **413** (`code: "context_too_large"`); `user_prompt` is truncated to 2000 chars; `candidates` clamped to 50.

**Rate limiting (Redis, per authenticated user)**

- Implemented as per-minute counters keyed by **`subscription_tier`** (`users.subscription_tier`: `free` \| `super` \| `premium` \| `enterprise`), **`task`** (e.g. `chat_topics`, `planner_theme_plans`, `winkly_plan`), and **`user.id`**.
- **`free`** tier has low per-minute ceilings for AI tasks when Redis is enabled (tune in `ai-gateway` as traffic grows).
- **`super`** / **`premium`** / **`enterprise`** get higher per-minute ceilings (tune in `ai-gateway` as traffic grows).
- On exceed: HTTP **429** with `retry_after` (seconds).

**Retries & quota handling (Gemini / OpenAI)**

- Gemini `generateContent` calls use **exponential backoff + jitter** and honor **`Retry-After`** when present.
- **Quota exceeded** (Gemini `RESOURCE_EXHAUSTED`, OpenAI `insufficient_quota`) is detected and treated as **non-retryable**: the gateway skips wasteful retries and returns HTTP **429** with `code: "quota_exhausted"`, `quota_exhausted: true`, and a longer `retry_after`. Transient provider 429s are still surfaced as `code: "rate_limited"`.

**Cost optimization**

- **Model routing**: topic/light tasks vs final structured plan use different model env vars (`GEMINI_MODEL_TOPICS` vs `GEMINI_MODEL_PLAN`).
- **Semantic cache (24h)**: e.g. `chat_topics` and `planner_theme_plans` responses cached by normalized keys (city/mode/theme/participants).
- **Participant context cache (24h)**: aggregated planning profile rows cached per conversation/participant set to reduce repeated DB + token work on refinements.

**Grounding control (Maps / Places cost)**

- `generateWinklyPlan` supports **`maps_grounding`**: **`none`** \| **`textsearch`** \| **`verify`**.
  - **Preview / two-option theme flow**: typically **`textsearch`** (single Text Search; lighter).
  - **Final `winkly_plan` (user confirms)**: **`verify`** (Text Search + Place Details for booking-style URLs where applicable).

**GCP / quota (operations)**

- Use **one** Google Cloud project; enable **Generative Language API** (Gemini), billing, and request higher RPM/TPM for production.
- Application-level Redis limiting smooths bursts; align Edge concurrency and Supabase plan with expected peak.

| **profile_photo_verifications**         | user_id, status (`photo_verification_status`), selfie path, profile_photo_index, provider, similarity_score, raw_response.                                                                                                                                                                                                                                                                                                                                                             |
| **date_safety_checkins**                | user_id, optional partner_user_id / planner_item_id, `date_checkin_status`, scheduled_at, checkin_due_at, responded_at.                                                                                                                                                                                                                                                                                                                                                                |

**compatibility_scores** (extended): optional **`behavior_affinity`**, **`ml_rank_score`** for hosted ML (e.g. SageMaker) and batch sync.

### 4.3 RLS (Row Level Security)

- **users**: SELECT/UPDATE own row only.
- **profiles_core**, **profiles_mode**, **profiles_business**: full access to own row(s).
- **follows**: SELECT if follower or followee; INSERT/DELETE as follower.
- **events**: SELECT if creator or participant; INSERT/UPDATE as creator. **event_participants**, **event_invitations**: participant or event creator.
- **planner_items**: SELECT if creator or in planner_participants; INSERT/UPDATE as creator. **planner_participants**: participant or item creator.
- **conversations**: SELECT if member; INSERT as created_by. **conversation_members**: member or conversation creator as per policies. **messages**: SELECT if member (history/evidence retained even after unmatch); **INSERT** requires `auth.uid() = sender_id` **and** active membership (`left_at IS NULL`) **and** **`is_dm_send_allowed(conversation_id, auth.uid())`** — i.e. only matched users can message (romance match DMs need an active mutual like; any block disables messaging). **No UPDATE/DELETE policy** — messages are immutable once sent (also enforced by trigger `enforce_message_immutability` for `authenticated`/`anon`).
- **message_read_receipts** / **message_delivery_receipts**: SELECT if conversation member; INSERT own rows only (`auth.uid() = user_id`).
- **groups**, **group_members**: creator or member.
- **wishlist_items**, **user_preferences**, **calendar_connections**, **ai_requests**: own rows only.
- **user_push_tokens**: own rows only (device tokens for Expo push).
- **user_locations**: own row only (SELECT/INSERT/UPDATE/DELETE where `auth.uid() = user_id`). **No policy exposes another user's coordinates** — distances are computed inside SECURITY DEFINER functions and returned rounded.

All tables in the schema have RLS enabled; no table is globally readable/writable by anon.

### 4.4 Triggers and RPCs

- **handle_new_user**: On INSERT/UPDATE of `auth.users`, upsert `public.users` with `id`, `email`, `account_type` from `raw_user_meta_data`.
- **create_direct_chat**: RPC to create or return existing DM between two users in a mode (p_user_a, p_user_b, p_mode, p_source, p_initiator). Sets/backfills **`dm_pair_key`** so there is exactly one DM conversation per matched pair per mode.
- **is_dm_send_allowed(conversation_id, user_id)**: SECURITY DEFINER helper used by the `messages` INSERT policy. Returns true for non-DM chats; for DMs returns false if a block exists either way; romance **match** DMs additionally require an active mutual `romance_like`.
- **enforce_message_immutability**: BEFORE UPDATE/DELETE trigger on `messages` that raises for `authenticated`/`anon` roles (evidence trail). Service-role and account-deletion cascades are unaffected.
- **mark_messages_delivered(p_message_ids[])**: SECURITY DEFINER RPC; inserts `message_delivery_receipts` for the caller's _received_ messages (skips own messages), filling `conversation_id`.
- **create_event_chat**: RPC to create or return event-linked conversation for an event.
- **Direct chat on mutual follow**: Triggers (see chat migrations) create DM when mutual follow is established where applicable.
- **Direct chat on mutual romance like**: **`trg_mutual_romance_like_create_dm`** (migration **`20260615120000`**) creates the romance match DM on the INSERT that completes the pair; **`romance_like_profile`** resolves **`chat_id`** from **`conversations.dm_pair_key`** after insert.
- **apply_message_behavior_signal**: After INSERT on **messages**, for DM in romance/friends/business, increments **`message_count`**, merges **`interaction_signals`** (character volume, reply-latency sample vs last peer message), and recomputes **`affinity_score`** via **`compute_behavior_affinity_score`**.
- **record_pair_behavior_signal**: Authenticated RPC for **`concierge_match_session`**, **`planner_from_chat`** (optional `conversation_id` in payload), **`invite_accepted`**, **`dm_first_outreach`** — merges into **`interaction_signals`** and refreshes **`affinity_score`** without requiring a new message row.
- **get_behavior_affinities(p_user_id, p_candidate_ids[], p_mode)**: Returns candidate → affinity for discover ranking (authenticated).
- **set_my_location(p_lat, p_lng, p_precision DEFAULT NULL)**: SECURITY DEFINER RPC; validates coordinates, **quantizes them server-side** (`_winkly_snap_coord` — `approximate` ≈1.1km / `precise` ≈110m grid) and upserts the caller's `user_locations` row (builds the `geography` server-side). **Raw GPS is never stored.** Precision falls back to the user's stored `user_profiles.location_precision` when `p_precision` is NULL. Clients never write the geography column directly.
- **set_my_location_precision(p_precision)** / **get_my_location_precision()**: SECURITY DEFINER RPCs to change/read the caller's discovery precision (`'precise'`/`'approximate'`). Coarsening re-snaps any stored point immediately; switching to precise applies on the next refresh.
- **romance_discover_feed_geo(current_user_id, p_max_distance_km, p_age_min, p_age_max, p_genders[], p_limit)**: SECURITY DEFINER (enforces caller `auth.uid()`). Romance discovery with **server-side distance / age / gender** filtering; excludes already-liked, **passed (`user_swipes`, mode='romance')**, and blocked users. Returns profile fields plus a **rounded** `distance_km` and `distance_label` (`"~3 km away"` / `"< 1 km away"`) — **never raw coordinates**. Caller without a stored location skips the distance filter; candidates without a location are kept (sorted last).

### 4.5 Views / compatibility

- **profiles_mode** may be backed by or aligned with **sub_profiles** (migration `align_sub_profiles`). App uses mode-scoped profile reads via `lib/access/profiles.ts` (`profiles_mode` for personal, `profiles_business` for business).
- **dates** (`20260608120000_dates_view.sql`) — read-only view over `planner_items` (`source_mode='romance'`, non-null `related_user_id`) LEFT JOIN `planner_invitations`, exposing `id, proposer_id, invitee_id, title, starts_at, ends_at, activity, location, place, invitation_id, status, created_at`. `security_invoker = on` so the caller only sees dates they're part of (underlying RLS). Romance dates are **not** a separate table — this view is a clean query surface over the existing planner model.

## 5. Authentication and session

### 5.1 Auth provider

- **AuthProvider** (`providers/AuthProvider.tsx`): Single source of truth for session.
- Uses `supabase.auth.getSession()` on load and `onAuthStateChange` for updates.
- Exposes: `session`, `user`, `loading`, `accountType` (from `user_metadata.account_type`), `signOut()`.
- Recoverable auth errors (e.g. invalid refresh token, session expired): clear session locally and optionally call `signOut({ scope: "local" })` so user is sent to splash.

### 5.2 Auth flow (high level)

1. **Splash** → if no session: **Welcome intro** (first launch) or **Sign in** (returning visitor; footer links to **get-started** for sign up). If session exists and **last activity** (local `winkly_last_activity_at` or Supabase `last_sign_in_at`) is within **14 days** and email is confirmed: **mode-selection** when core profile is complete, else **welcome-back-setup**; if activity is older than 14 days, local session is cleared → **Sign in** (**Welcome back** when `winkly_has_account`). Logic: `lib/routing/splash.ts`, `lib/lastActivity.ts`.
2. **Sign up / Sign in**: email + password, or **Google** / **Apple** OAuth (`OAuthButtons`, `lib/auth/oauth.ts`). New OAuth users get `account_type` in user metadata (sign-up passes personal/business). Sets `winkly_has_account`. Production: configure providers in Supabase Auth + EAS env per **docs/GO_LIVE_AND_PLAY_STORE.md** §1.4.
3. **Email verification**: Supabase sends link. Link can point to **auth-redirect** (Edge Function or Vercel) that redirects to `winkly://callback` with hash/query so in-app browser can open the app.
4. **Callback** (`(auth)/callback.tsx`): Handles deep link. If **terms-cookies** not yet accepted, stores the URL in **`pendingAuthCallback`** and routes to **`/(auth)/terms-cookies?next=callback`** before **`setSession`** (GDPR). After acceptance, resumes the stored URL and exchanges fragment for session.
5. **Post-auth**: RouteGuard + ModeContext load; if no mode selected or no permissions, user goes to **mode-selection** or onboarding.

### 5.3 Auth redirect (email verification)

- Problem: Gmail/in-app browsers may not open `winkly://` directly; user sees white page.
- Solution: Redirect URL in Supabase set to an HTTPS page that:
  - Shows “Opening Winkly…”
  - Redirects to `winkly://callback` with hash/query (tokens).
  - Fallback: “Tap here to try again” link with same deep link.
- Implementations:
  - **Supabase Edge Function** `auth-redirect`: Serves HTML with correct `Content-Type: text/html`. Deploy with `npm run supabase:deploy-auth-redirect`. Set **`AUTH_REDIRECT_STATE_SECRET`** in Edge secrets (prod). Mobile uses **`getEmailRedirectTo()`** for `emailRedirectTo`. Add function URL to Supabase Redirect URLs.
  - **Vercel**: Deploy `auth-redirect/`; add that URL to Redirect URLs; set `EXPO_PUBLIC_AUTH_REDIRECT_URL` in app.
- Config: `constants/config.ts` uses `EXPO_PUBLIC_AUTH_REDIRECT_URL` or default `winkly://callback`.

### 5.4 Password reset

- **Flow (same as sign-up confirmation)**: User requests reset → receives email → taps link → link opens app via deep link (or auth-redirect) → **callback** establishes session → **reset-confirm** to set new password.
- **Request**: Sign in → "Forgot your password?" or Account → reset-password. `resetPasswordForEmail(email, { redirectTo: AUTH_REDIRECT_URL })` sends the reset link.
- **In app**: Callback handles the link (fragment or query); if recovery type, routes to **reset-confirm**. User sets new password then returns to sign in.
- **Fallback (Expo Go / dev)**: Paste-reset-link block on reset-confirm is shown only in `__DEV__` (like verify.tsx). User can reach reset-confirm via "Already have the link? Set new password" (after sending) or "Already have a reset link?" (sign-in). App accepts redirect URL with fragment or initial verify URL with `?token=...&type=recovery` via `verifyOtp`.

### 5.5 Session persistence

- Supabase client uses `AsyncStorage` for auth storage, `persistSession: true`, `autoRefreshToken: true`, `detectSessionInUrl: false` (no URL-based session detection in RN).

### 5.6 Auth trigger (public.users)

- `handle_new_user()` syncs `auth.users` → `public.users` (id, email, account_type). Trigger runs **only on INSERT** (migration `20250306100000_auth_trigger_insert_only.sql`) so that login (which updates `auth.users`) does not run the trigger and cannot cause "Database error granting user".

## 6. Identity Firewall and mode context

### 6.1 ModeContextProvider

- **ModeContextProvider** (`providers/ModeContextProvider.tsx`): Single source of truth for **active mode** and **permissions**.
- Loads from `users` (account_type, is_premium, subscription_tier) and `sub_profiles` / `profiles_mode` (which modes exist for this user). Builds `permissions: Mode[]` (always includes `events`; for personal adds romance/friends/business if sub-profile exists; for business adds business).
- Context shape: `user_id`, `account_type`, `active_mode`, `active_persona_id`, `permissions`, `subscription_tier`. In dev, `subscription_tier` is forced to `premium` for testing.
- **Caching (stale-while-revalidate, 60s):** a **module-level in-memory cache** keyed by user id holds the DB-derived fields (`account_type`, `permissions`, `subscription_tier`). On load, a cached value is applied **immediately** (no spinner) and only re-fetched when older than `CONTEXT_CACHE_TTL_MS` (60s) or via **`refresh()`** (force). Network/parse errors keep the existing cached context; the Events-only fallback only applies on a fresh start. The cache is cleared on sign-out / no user. The provider exposes **`refresh(): Promise<void>`** alongside `setActiveMode` / `resetMode`; `mode-selection` calls it on focus so a freshly-completed sub-profile re-syncs permissions before mode entry.

### 6.2 RouteGuard

- **RouteGuard** (`components/RouteGuard.tsx`): Runs after Auth + ModeContext load.
  - No session and not on auth route → replace to `/(auth)/splash`.
  - Session and on a mode route (romance/friends/business/events) and `context.active_mode` set but not in `context.permissions` → replace to `/(tabs)/mode-selection`.

### 6.3 Mode switching

- **setActiveMode(mode, personaId?)**: Updates context and **replaces** router to the corresponding mode root: `/(modes)/romance`, `/(modes)/friends`, `/(modes)/business`, `/(modes)/events`. This avoids stacking mode-specific screens from another mode (Identity Firewall).

### 6.4 Access layer (lib/access)

- All mode-sensitive data access goes through `lib/access/*`: `profiles`, `planner`, `events`, `conversations`, `connections`.
- **getProfilesForMode(mode, userId)**: Uses `profiles_mode` for romance/friends, `profiles_business` for business; no cross-mode reads.
- **getProfileForMode(mode, viewerId, targetUserId)**: Single-user read for profile-view entry points (matches, discover cards, chat avatars). Romance → `public_profile_view`; Friends → `friend_profiles`; Business → `business_profile_for_viewer` RPC (business accounts) or `profiles_mode` + `user_profiles` join (personal business sub-profile).
- **getPlannerItems(userId, sourceMode?)**: Optional filter by `source_mode`.
- **getConversations(userId, mode?)**: Mode-filtered; RLS ensures only member conversations.
- **getEventsForUser**, **getEvent**: RLS enforces participant/creator access.

### 8.4 Chats

- **Types**: DM, group, event, system, AI (ConversationType). Display: direct, group, event, system.
- **API** (`lib/chats/api.ts`): sendMessage (returns normalized **Message** row), createDirectChat (RPC), createEventChat (RPC), blockUser, unmatchRomance, unfollowConnection, reportUser, reportMessage, markMessagesAsRead (seen), **markMessagesDelivered** (delivered, via RPC), getReadReceipts/getDeliveryReceipts, etc. Messages support text, attachments, reply_to_id, message_type (including **icebreaker** — JSON payload with prompt). **No delete APIs** — messages are immutable post-send (safety / evidence trail); misuse is handled via **report + block**.
- **Realtime**: `useMessageSubscription` listens for **INSERT** on `messages`; payloads are normalized (`lib/chats/normalizeMessage.ts`) and merged into local state (dedupe by `id`). Subscription handler is kept stable via a ref so channels are not recreated every render. Relevant tables (`messages`, `typing_indicators`, `message_read_receipts`, `message_delivery_receipts`) are in the **`supabase_realtime`** publication.
- **Optimistic UI**: outgoing messages render **immediately** as a pending bubble (lowered opacity + "Sending…") before the server INSERT resolves. Each carries a locally generated **`client_id`** (also persisted on the row, migration `20260612120000_messages_client_id.sql`) so the authoritative row — whether it arrives via `sendMessage`'s return or Realtime — **replaces** the placeholder with no duplicates (`mergeIncomingMessage` reconciles by `client_id`, then `id`). ### 8.4a Discover (Romance & Friends)

- **Purpose**: Discover is high-probability matches and curated recommendations, not random swiping. **Romance & Friends** screens have **no page-level subtitle** (the bottom tab labels the hub) and **no header filter button**; discovery preferences are edited on **Home** (`/(modes)/romance/filters`, `/(modes)/friends/filters`). Tone: calm, intentional, exclusive.
- **Layout**: Vertically scrollable page of **five horizontal rows** (`DiscoverHorizontalSection`). Each row scrolls left/right; profile cards show **name + age on the photo** (bottom overlay). Per-row empty copy: _Unfortunately at the moment there are no people in this category. Check later._
- **Categories (Romance titles / Friends equivalent)**:
  1. **Liked you (N)** / **Want to connect (N)** — all incoming likes/follows-not-reciprocated. Romance: `romance_likes_received`. Friends: `follows` where followee = me and I do not follow back.
  2. **Recommended by Winkly** — up to **4 new profiles/day** (`lib/discover/sectionStorage.ts`); Romance pool from **`romance_discover_feed_geo`** (+ client filters + optional AI sort); Friends from **`friends_discover_feed`**.
  3. **Same Interests** — up to **4 new/day**; overlap with viewer's interests; Home filters applied.
  4. **Same dating goals** (Romance) / **Same meetup goals** (Friends) — up to **4 new/day**; overlap on `relationship_goals` / `meetup_goals` in mode meta; Home filters applied.
  5. **New people nearby** — up to **4 new/day**; Romance sorted by `distance_km` from geo feed; Friends prioritises same city when known.
- **Friends incoming requests** (`friends_requests`): Super Connect / connect rows targeting the current user are listed at **`/(modes)/friends/friend-requests`** (Friends Home banner when pending; **Chats → New conversation → Friends connection requests**). **Accept** → **`friends_accept_request`** (mutual **`follows`**, row removed; opens DM when one exists). **Decline** → **`friends_decline_request`**. Migration **`20260606140000_friends_requests_accept_decline.sql`**.
- **Filters**: Categories **2–5** apply saved Home filters via `applyRomanceFiltersToFeed` / `applyFriendsFiltersToFeed` (and Romance geo RPC params for distance/age/gender). Feed excludes users already liked/passed and any **block relationship** (`user_blocks`). When **AI matching** is on, Romance/Friends pools sort with **65% compatibility + 35% behavior affinity** (`get_behavior_affinities`).
- **Romance Home (stack swipe)**: Card-stack deck (right = like, left = pass; up/tap = view profile; star = super-like). Loads via **`fetchRomanceSwipeDeckProfiles`** → RPC **`romance_discover_feed_geo`** with the saved filters (max distance, age range, **"Show me"** gender preference) so **distance / age / gender** are filtered **server-side**; passes (`user_swipes`) and blocks are excluded; optional AI behavior-sort still runs client-side. Each card shows a **rounded distance label** (`MatchCardOverlay`, from `distance_km`/`distance_label` — never raw coordinates). On a mutual like the **`MatchCelebration`** overlay plays an animated **"It's a match!"** moment (both avatars, floating hearts) with **Send a message** (→ DM, `?matchBridge=1`) / **Keep swiping**. Falls back to legacy `romance_discover_feed` then `public_profile_view` if the geo RPC is unavailable. **Empty deck:** **`SwipeDeckEmptyState`** — copy nudges expanding distance; **Expand search radius** → filters; likes-received count teaser → Discover.
- **Friends Home (stack swipe)**: Same card-stack pattern as Romance; loads via **`fetchFriendsSwipeDeckProfiles`** → **`friends_discover_feed`**. **Empty deck:** same **`SwipeDeckEmptyState`** with want-to-connect count teaser → Discover.
- **Location for distance**: Coarse location is refreshed when permission is already granted (Romance Home focus, throttled 6h) via **`lib/location/updateLocation.ts`** → RPC `set_my_location`. **First time in each mode**, **`ModeLocationGate`** shows an in-app rationale before the OS prompt; declining skips location until the user enables it in settings. Coordinates are **quantized server-side** to the user's chosen precision (`approximate` ≈1.1km default / `precise` ≈110m) so **raw GPS is never stored**; precision is configurable in **Privacy & Safety → Location precision** (`app/account/location-privacy.tsx`). Stored in **`user_locations`** (PostGIS), owner-only; other users only ever see the rounded distance label. See §10.1.
- **Subscription**: **Free** — first **3** cards visible in **Liked you** and **Recommended by Winkly**; additional cards blurred; tap blurred → **`DiscoverUpgradeModal`** → **`/account/subscription`** (Super/Premium). Free users can match only with non-blurred cards in those rows. **1** like/connect per day from Recommended. **Super/Premium** — full row visibility; unlimited recommendation likes.

### 8.8 AI

- **Concierge rollout — Phase A (current):** AI planning targets **1:1 chat** (DM: `partner_user_id` + `partner_display_name` on `/concierge`) and **Planner** (solo or one invitee via `ConciergePlanningFlow` / `ConciergeRequestForm`). **ai-gateway** `getConciergeProfileContext` loads **`profiles_core`** (general) + **`profiles_mode` for the request `mode` only** for primary and optional partner; **`summarizeModeProfile(mode)`** includes only goals relevant to that mode (Romance: relationship goals; Friends: meetup goals; Business: networking goals + `professional_topics`). **Phase B (planned):** group chats — multi-select members (cap TBD), same rule per participant (**general + Friends or Business sub-profile for that thread’s mode**; never Romance data in a Friends group context, etc.).
- **Two-layer architecture (scalable):** **(1) Precomputed compatibility (Layer 1):** No LLM. Stored in `compatibility_scores` (user*a_id, user_b_id, mode, compatibility_score 0–1, shared_interest_tags, shared_activity_tags, budget_overlap, location_proximity_bucket, confidence_score; optional **behavior_affinity**, **ml_rank_score**). Optional `profile_embeddings` (pgvector) for similarity. Updated by Edge Function `recompute-compatibility` (trigger: profile/interest/preference change or cron; body: user_id?, mode?). **DM behavior** is captured in `behavior_pair_signals` (trigger on messages). Edge **`recompute-behavior-ml`** (POST with **Authorization: Bearer `SUPABASE_SERVICE_ROLE_KEY`** only) merges behavior into `compatibility_scores` and optionally calls **`SAGEMAKER_RUNTIME_ENDPOINT`**; optional \*\*`UPSTASH_REDIS_REST*\*`** cache bust per user. **(2) On-demand LLM + cache (Layer 2):** LLM only when user takes high-value action (e.g. Plan date, Suggest another plan). When client sends `compatibility_context`(from Layer 1), gateway uses compressed prompt (COMPATIBILITY_SUMMARY) instead of full profiles. Results cached in`ai_plan_cache`(24h TTL); reuse when same chat reopened. Client:`lib/ai/compatibilityLayer.ts`.
- **Client heuristics** (no LLM in app): `lib/ai/romanceInsights.ts` (compatibility score, match tags, conversation starter, profile insight from Romance profile fields); `lib/ai/friendsInsights.ts` (Friends compatibility, match tags). Deterministic, no PII sent to external AI in these.
- **AI Gateway** (Supabase Edge Function `ai-gateway`): Authenticated; accepts mode, task, allowlisted context (including partner_user_id, refinement_feedback, previous_options, **primary_free_slots** / **partner_free_slots** for match bridge, **`conversation_id`** / **`target_slot_iso`** / **`search_radius_miles`** for Match Agent). Task **`match_agent`** (Romance or Friends): pipeline (profiles + planner items, Places midpoint search, weather, LLM) returns **`match_agent`** payload and optional **`proposal_id`** (`ai_match_agent_proposals`). Task **`match_bridge`** (Romance only): builds PRIMARY_USER / PARTNER_USER, **PRIMARY_PLANNER_ITEMS** / **PARTNER_PLANNER_ITEMS** (service role), WINKLY_EVENTS_CANDIDATES / WINKLY_BUSINESS_CANDIDATES / EXTERNAL_PLACE_HINTS; returns JSON **match_bridge** (bridge_message, proposed_starts_at, venue_name, …). **Premium/Enterprise:** Anthropic (Claude) primary; **Super:** Gemini primary; OpenAI/Gemini fallbacks; else **fallback** text. For plan/concierge/event_suggest: fetches primary and optional partner profile (profiles_core + profiles_mode) and injects PRIMARY_USER, PARTNER_USER, LOCATION. Profile data sent to the AI is minimal for personalization only: age, gender, city, interests, bio (truncated), lifestyle, dietary/allergies, values, goals, transport; first name, last name, and email are never sent. The system prompt instructs the model to use this profile data when making suggestions (e.g. match activities to interests, consider age and gender-inclusive options, respect dietary and location). Elite Concierge system prompt (4-step reasoning, mode weighting, 3-option JSON). **Before** the LLM call, the gateway builds **WINKLY_EVENTS_CANDIDATES** from public **`events`** (keyword + city scoring merged with optional RPC **`match_events_for_concierge`** using `pg_trgm` when migration `20260406120000_concierge_event_trgm_and_rpc.sql` is applied). Injects **WINKLY_BUSINESS_CANDIDATES** from `business_services` + `profiles_business` (keyword/city scoring). Injects **EXTERNAL_PLACE_HINTS**: Google Places Text Search when **`GOOGLE_PLACES_API_KEY`** or **`GOOGLE_MAPS_API_KEY`** is set; else OpenStreetMap Nominatim (rate-limited). The model prioritizes Winkly rows over external hints for bookings. Client builds **plan_request_text** (same facts as typing into Gemini: topic, place, dates, time, budget, weather line). When **weather_snapshot** is sent, the gateway **does not** expose the `get_weather` tool (avoids duplicate calls; weather is the app’s forecast for that location/dates). Remaining tools: get_winkly_events, get_planner_items (and get_weather only when no snapshot). **Providers:** Premium/Enterprise → `ANTHROPIC_API_KEY` (Claude) first, then Gemini, then OpenAI; Super → Gemini first, OpenAI fallback. Returns message + parsed suggestions (Experience Menu). See **docs/AI_CONCIERGE_SPEC.md**, **docs/AI_CONCIERGE_RECOMMENDATION.md**, and **docs/DATA_PROTECTION_AND_PRIVACY_ASSESSMENT.md** (AI section).
- **Concierge client** (`lib/ai/conciergeClient.ts`): `callConcierge({ task, context })` for plan/concierge/event_suggest/**match_agent**; context allowlisted (same fields plus **conversation_id**, **target_slot_iso**, **search_radius_miles** for Match Agent). **`match_agent`** response includes **`match_agent`** (agent_message, draft, chain, proposal_id, privacy). Otherwise response includes suggestions (ExperienceOption[]). Full spec: **docs/AI_CONCIERGE_SPEC.md**.
- **Concierge agency (2026):** **`user_concierge_signals`** (JSONB: avoid/prefer venues, noise_level, professional_topics) merged in **ai-gateway** with profiles into **PREFERENCE_ENGINE_MERGE**; device calendar white space via **`getMergedDeviceWhiteSpaceSlots`** → **`calendar_white_space`**; OpenTable/Resy **discovery** URLs via **`booking_context`** (`lib/integrations/bookingLinks.ts` — not confirmed bookings). **Stale DM nudge** (Friends/Business, last message **48h+**): RPC **`conversation_eligible_for_concierge_nudge`**, dismiss **`dismiss_concierge_nudge`**; chat banner → Concierge with **`prefill_prompt`**. **Google Calendar** cloud merge: Edge **`calendar-freebusy`** (stub until OAuth refresh wired); **`calendar_connections`** extended with `scopes`, `token_expires_at`. Architecture (LangChain vs Edge tools, Pinecone vs pgvector): **docs/AI_CONCIERGE_AGENCY_ARCHITECTURE.md**.

Display format is **City, Country** (comma and space). **`expandCountryForDisplay`**: Resolves ISO 3166-1 alpha-2 (DE, UA, GB, …) via `Intl.DisplayNames` in the app language; maps common non-ISO tokens (**UK→GB**, **USA→US**, **UAE→AE**) and a subset of **ISO alpha-3** codes (e.g. UKR→UA, DEU→DE) before expanding. Already-full names pass through unchanged. **`formatDefaultLocationDisplay`**: If the `city` argument contains a comma, it is treated as a full “City, Country” line and normalized; the separate `country` argument is not appended again (avoids `Olching, DE, Germany`). Hooks in `lib/location/useLocationDisplay.ts`: `useFormatLocationDisplay()` (callback), `useNormalizedLocation(loc)` (memo) — prefer these in screens that show many city lines. Use the **app language** from `useTranslation()` where UI is shown (default English otherwise). **Sources**: Open-Meteo autocomplete (`searchLocationAutocomplete` in `weatherClient.ts` normalizes each suggestion); **`reverseGeocodeToDisplay`** in `deviceLocation.ts` (also used by profile-core **Use my location**) expands ISO to full country names. **`useDefaultLocation`** (`lib/ai/useDefaultCity.ts`): Prefers **`user_profiles.city`** (onboarding/account), then **`profiles_core`**; splits/normalizes combined lines. **App surfaces** (non-exhaustive): Concierge saved ideas + forms + Activity details + summary step; Planner AI Custom plan pre-fill + planner item details modal; `EventCard`; Romance/Friends discover `DiscoverRecommendedSection`; chat planner-invite CTA bubbles (`chat-view`); `InviteToPlanModal` + `SuperLikeInviteModal` city/area pre-fill (profile defaults); `profile-core` + `profile/edit-core`; Romance liked/matches; Chats new-chat; Events discover/details/share; Business companies; `EventParticipantCard`; Chats matches strip + inbox; discover `MatchCardOverlay`; mode profile views; profile preview; `useDefaultLocation` / `useDefaultCity`.

## 10. External integrations

### 10.1 Device calendar and location

- **expo-calendar**: Read/write device calendar (Apple Calendar / Android calendar). No Supabase change required. **Opt-in only:** writing planner items to the device calendar is gated by the user-facing "Sync planner items to calendar" toggle in `planner/settings` (preference `winkly_planner_calendar_sync_enabled`, defaults OFF). Use `isCalendarSyncEnabled()` (`lib/integrations/calendarSync.ts`) before any calendar write — it requires both the opt-in toggle and a granted OS calendar permission. (Read-only `getEventsAsync` for AI free-slot detection is unaffected.)
- **expo-location**: Foreground location for event locations and discovery. **Permission UX:** each mode's first launch shows an in-app rationale (`ModeLocationGate` / `ModeLocationRationaleModal`) before `requestForegroundPermissionsAsync`; planner settings and profile onboarding follow the same contextual pattern. For Romance geo-discovery the coarse point is refreshed when granted (throttled 6h) via `lib/location/updateLocation.ts` → RPC `set_my_location`, stored in **`user_locations`** (PostGIS). Raw coordinates are **owner-only**; distance is computed server-side and returned rounded (`romance_discover_feed_geo`). GDPR: no continuous tracking, no exposure of others' coordinates.

### 10.2 Google Calendar cloud sync (future)

- **calendar_connections** table: user_id, provider, external_id, token_encrypted, last_sync_at. RLS: own rows only.
- Flow: Planner settings “Allow calendar access” → OAuth via Edge Function → store tokens → sync planner_items to Google Calendar (and optionally reverse). Secrets (Google client id/secret) in Edge Function only. See `docs/SUPABASE_CALENDAR_MAPS_INTEGRATION.md`.

### 10.3 Maps / directions

- Store location, location_lat, location_lng on planner_items and events (migrations may add lat/lng if missing).
- Open directions: `Linking.openURL` with Google Maps or Apple Maps URL. No Supabase setup for opening maps.

### 10.4 AI and weather (Edge Function)

- **ai-gateway**: `ANTHROPIC_API_KEY` (Premium primary), `GEMINI_API_KEY`, and/or `OPENAI_API_KEY` in Supabase secrets; Open-Meteo for weather/geocoding (no key). No client-side API keys. Free-tier 429s are mitigated by cross-provider fallback and retries; paid keys raise limits. **Production:** optional Upstash Redis, topic vs plan models, Places grounding modes — **§3.7** and **§11.3**.
- **Future**: Google Weather API or Google Places can be added as tools in the same Edge Function; keys in Supabase secrets only.

### 10.5 Expo Push (notifications)

- **Client**: `expo-notifications` + `lib/notifications.ts` — initializes foreground handler, Android **`default`** channel, requests permission on login, **`getExpoPushTokenAsync`** (uses `expo.extra.eas.projectId` from **app.config.js**), upserts into **`user_push_tokens`**. Also exposes `getInitialNotificationData` / `addNotificationResponseListener` for tap deep-linking (`NotificationDeepLinkHandler`).
- **Server-side fan-out — `notify-fanout` (messages + matches)**: the **primary** path for chat-message and new-match pushes. Postgres **AFTER INSERT** triggers on `messages` and `romance_likes` (migration `20260612130000_push_notification_triggers.sql`) call this Edge Function via **pg_net** the moment the row commits, so delivery does not depend on the sender's app staying alive. It resolves recipients (active `conversation_members` − sender − **muted** via `conversation_member_settings`) / the matched peer, loads `user_push_tokens`, and sends Expo pushes. Auth: **`x-webhook-secret`** header == function env `WEBHOOK_SECRET` (`verify_jwt = false`). The match trigger fires only when the like becomes **mutual** and includes `chat_id` for deep-linking (DM is created by **`trg_mutual_romance_like_create_dm`**, migration **`20260615120000`**, which runs first). Triggers **no-op** until `private.webhook_config` (`function_base_url`, `secret`) is populated.
- **Edge `expo-push-notify`** (now **planner** kinds): Authenticated user POSTs `kind` (`planner_invitation` \| `planner_response`; `new_match`/`chat_message` still supported but no longer called by the client), `recipient_user_id`, `title`, `body`, optional `conversation_id` / `planner_invitation_id`. Server verifies relationship before calling Expo’s **`/push/send`** API: a **pending** invite where caller=inviter, recipient=invitee (`planner_invitation`); or a **non-pending** invite (`planner_response` — notifies the proposer that their date was accepted).
- **Secret**: **`EXPO_ACCESS_TOKEN`** (Expo account) recommended on the Edge runtime for reliable delivery.
- **Plan confirmed**: **`pending-plan-confirm`** loads tokens for all `participant_ids` and sends **Plan confirmed** when the planner item is finalized.

### 10.6 Deep linking

- Scheme: `winkly://`. Callback path: `winkly://callback` for auth. Used after email verification and OAuth redirects.

### 10.7 Storage (media buckets)

- **Buckets** (migration `20260613120000_storage_buckets_policies.sql`, mirrored in `supabase/config.toml`): **`user-photos`** (profile/core/sub-profile photos, chat images, verification selfies), **`user-videos`** (profile/sub-profile videos + chat/profile voice notes/audio), **`business-logos`**. All are **public-read** (so `getPublicUrl()` works for media display) with owner-scoped write RLS on `storage.objects`: the first path segment must equal `auth.uid()` (paths are `{userId}/{mode}/{file}`).
- **Limits enforced backend-side:** `file_size_limit` **50 MiB** (52428800 bytes) per bucket and `allowed_mime_types` per bucket (images: jpeg/png/webp/heic/heif; videos: mp4/quicktime; audio: mp4/mpeg/aac/x-m4a; logos: jpeg/png/webp).
- **Client-side validation (avoids wasted uploads):** `apps/mobile/lib/mediaValidation.ts` (`MAX_UPLOAD_BYTES`, `validatePickerAsset`, `validateMediaForUpload`) checks size + MIME **before** any upload; wired into `lib/uploadMedia.ts`, `lib/uploadLogo.ts`, and onboarding `profile-core`.
- **Onboarding persistence:** `profile-core` now uploads local photos/videos to Storage on save (`uploadLocalPhotos` / `uploadLocalVideos`) and persists the resulting **public URLs** (not `file://` URIs) to `user_profiles`/`profiles_core`/`sub_profiles`.

## 12. Security and compliance

### 12.1 Technical security (summary)

- **No PII in AI telemetry**: `ai_requests` stores only user_id, mode, task.
- **Privacy Proxy (PII scrubbing)**: `supabase/functions/ai-gateway` scrubs obvious identifiers (email/phone/handles/URLs/address-like text) from free-text fields (`plan_request_text`, `user_prompt`, etc.) and from user-written bios before sending context to LLM providers. The gateway also caps `user_prompt` to **2000 chars** server-side to bound LLM costs and reduce prompt-injection surface.
- **RLS on all tables**: No table bypass; anon key used with strict RLS.
- **Mode isolation**: Access layer and RouteGuard prevent cross-mode data and navigation.
- **OAuth/secrets**: Only in Edge Functions or server-side; never in client.
- **Auth redirect**: HTTPS page for verification links to avoid in-app browser issues; deep link to app with tokens.

### 12.2 Data protection and legal

- **European and international**: User personal data is used and protected in line with GDPR (EEA/UK), ePrivacy considerations, and international practice. See **docs/DATA_PROTECTION_AND_PRIVACY_ASSESSMENT.md** for the internal assessment (data inventory, lawful basis, rights, transfers, recommendations).
- **Terms of Service**: **docs/TERMS_OF_SERVICE.md** — built and hosted at `https://winkly.app/terms` via **`website/`** (see `website/README.md`). Covers acceptance, use rules, account, subscription, content licence, AI, liability, governing law (Germany).
- **Privacy Policy**: **docs/PRIVACY_POLICY.md** — hosted at `https://winkly.app/privacy` (includes `#cookies` anchor for cookie notice). GDPR-aligned controller, data collected, purposes, legal basis, retention, rights, international transfers, security, contact.
- **Imprint (Impressum)**: **docs/IMPRINT.md** — hosted at `https://winkly.app/imprint` (TMG §5). Entity details substituted from **`website/legal-entity.json`** at build time — update before production deploy.
- **Community Guidelines**: **docs/COMMUNITY_GUIDELINES.md** — hosted at `https://winkly.app/community`.
- **Play Store Data safety**: **docs/PLAY_STORE_DATA_SAFETY.md** — declaration checklist aligned with Privacy Policy (location, photos, messages, PostHog analytics; no ads; data not sold).
- **First use**: On first app use, before sign-up or sign-in, users must accept **Terms & Conditions** and **Cookies** on a dedicated screen (`(auth)/terms-cookies`). Acceptance is stored locally; they can review Terms, Privacy, and Cookie notice at any time in Settings.
- **In-app**: General settings → Support & legal → **Legal** opens a single place for: Terms of Service, Privacy Policy, Cookies, Community Guidelines, **Data protection** (link to Privacy), **Imprint (Impressum)** (winkly.app/imprint), and Contact support. See `apps/mobile/app/account/legal.tsx`. URLs: winkly.app/terms, winkly.app/privacy, winkly.app/privacy#cookies, winkly.app/community, winkly.app/imprint.
- **Account deletion**: Implemented via Edge Function `delete-account` (Storage cleanup + auth.admin.deleteUser). In-app: General settings → Account & Identity → Delete or deactivate account → Delete my account. All user data (including `ai_requests`) is removed by CASCADE. AI gateway does not log full prompts; only (user_id, mode, task) in ai_requests, which is deleted with the account. Data export is not yet implemented.
- **Delete AI memory (without deleting account)**: Implemented via Edge Function `delete-ai-memory`. In-app: General settings → Privacy & safety → **AI memory**. Deletes the user’s `profile_embeddings` (vector profile) and AI caches/telemetry (and optionally `user_concierge_signals`) for one mode or all modes.
